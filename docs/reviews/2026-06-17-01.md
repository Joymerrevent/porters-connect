## Run 2 — RV-1〜4 fixed 検証＋RV-5 起票（HEAD ed50570）

初回で起票した RV-1〜4 が #33–35 等で fixed になった後の検証 run。RV-1（ADR-0020 実装）を中心に確認し、新規に RV-5 を起票。

**品質ゲート**: 全 6 緑。177 tests（初回比 +8。既定 field 送信を pin するテスト追加・RV-2 対応）。branch 98.22%。

**今回の指摘**: 新規 RV-5（🟡）。RV-1〜4 はゲート緑・実装・テストで fixed を検証。

**観点別所見**:

- **1. API 忠実性 — 🟡 RV-5（RV-1 は fixed 検証）**:
  - _RV-1（fixed・検証）_: ADR-0020（案A+2a+3a+軸4）が実装に正しく落ちている。`resource.ts:127-132` の `defaultFieldList` がカタログから既定 field を生成し、User 型は 4 サブ項目を入れ子展開、System[Reference] は `()` 省略で ID のみ＝`decode.ts` の `decodeUser`/`decodeReference` と 1:1。`search` は `field: query.field ?? defaultFields`（`:229`）で **省略=既定送信／`[]`=主キーのみ（`buildReadUrl:172`）／明示=verbatim** の 3 系統。`get` も内部 `search` 経由で既定 field を得る。Attachment は bespoke で `DEFAULT_FIELDS`（メタデータのみ・`Content` 除外）。Read クエリは `URLSearchParams` で URL エンコード。これで `one?.P_Name`（`README.md:63`）が実機で成立。
  - _Option 複数選択（再確認・良好）_: `decode.ts:82-91` の `decodeOption` は `Object.keys(root)` で**選択された全 alias を `string[]`** で返す（ADR-0017）。過去メモの「複数選択で先頭のみ＝データ損失」は解消済み。
  - _RV-5（新規）_: 送信前リクエスト長ガードは `requester.ts:75-87` で **`req.body` の長さのみ**を見る。GET（Read）は body を持たず `field`/`condition` は URL に載るため Read 経路は対象外。ADR-0020 で既定 field を全送信するようになり Read URL が伸びた。正典は「リクエスト**全体**の長さ ~15000 文字」（`docs/reference/resource-api/README.md:86`）＝GET では URL 全体が対象。超過時に明確な `PortersConfigError` でなく不透明な server 400 に倒れる。加えて ADR-0020 の Consequences(Neutral)（`0020-…md:83`）は「既存の送信前ガードが働く」と記すが Read 経路は未カバー＝**ADR の記述と実装が食い違う**。現行 MVP カタログは小さく（最大 Job=34・既定 Read URL ~1.2KB）**latent**だが、中核フェイルセーフの非対称な穴＋ADR 誤記として退行追跡に起票（→ findings RV-5）。
- **2. アーキテクチャ / ADR 遵守 — 問題なし**: 層分離維持。`resource.ts` は decode/encode を `xml/` から import し XML 構築を持たない。既定 field 生成は `defaultFieldList` としてカタログ駆動（ADR-0019 の SoT を一段拡張）。Attachment は ADR-0018 の bespoke どおり、値のエスケープは `xml/encode` の `encodeField` に委譲（タグ名は定数）＝許容範囲。
- **3. コーディング規約（ADR-0013）— 問題なし**: 新規 `defaultFieldList`/`bareAlias` とも arrow const、型は `type`。generator 例外はコメント明示。バレル／明示キュレーション維持。lint:ts 緑。
- **4. 型安全 / 公開面 — 問題なし**: `SearchQuery.field` の JSDoc（`resource.ts:44-55`）が「省略=既定／`[]`=主キーのみ／非空=verbatim」を明示し実挙動と一致。`src/index.ts` は内部実装を漏らさず必要な型のみ。公開 JSDoc は英語。
- **5. エラーモデル — 問題なし（RV-3 fixed 検証）**: `porters-error.ts:11-14` で `"rateLimit"` を**予約**と明記（コメント）。`README.md:241` も整合。`src/` 内で返す箇所は型定義のみ＝README 例との不整合は解消。
- **6. テスト厳密性 — 問題なし（RV-2 fixed 検証）**: 既定 field 送信を pin（`resource.test.ts:134-158`／`candidate.test.ts:83-92`／`attachment.test.ts:146`）。`ALL` fixture は既定 field 注入で「全項目要求」と整合し非現実的でなくなった。`decodeURIComponent` で復号後に検証＝URL エンコード実態と一致。Stryker equivalent-mutant 抑制は根拠コメント付き。
- **7. ドキュメント / DX — 問題なし**: README が ADR-0020 を反映（検索クエリ節 `:130-132`／get・P_Name 例 `:62-63`）。非公式明示・契約必須・Version 2・対応バージョン維持。md は参照スタイル。LV 整合。
- **8. リリース準備 — 問題なし（RV-4 fixed 検証）**: `publishConfig.access:"public"`／`repository`/`bugs`/`homepage`/`keywords`/`author`／`files`／`exports` 設定済み。生成物は未追跡。MVP リソース充足。`version:"0.0.0"` 妥当。

**良かった点**: RV-1 修正の質（カタログ→既定 field を decode 契約と 1:1、`field:[]` で API ネイティブも到達可能＝fidelity とフェイルセーフ両立・透明化）。退行を捕まえる体制（3 系統を個別テストで pin、169→177）。LV による不確実性管理継続。

**次アクション（当時）**: RV-5 は軽量なら ADR-0020 の文を訂正、堅くするなら `requester.ts` のガードを Read URL 長へ拡張。
