### 2. アーキテクチャ / ADR 遵守 — 🟡 RV-9

層分離（transport → requester → resources → xml）・依存方向（fields → resources・循環なし・RV-8 で整理済み）は維持。新規に accept された **ADR-0029（自動タグ＋OIDC publish）/ ADR-0030（手動 back-merge）/ ADR-0031（版番号検証）** の実装は概ね決定どおり（`tag.yml`・`release.yml`・`scripts/check-release-invariants.mjs`・`ci.yml` の `fetch-tags`）。

ただし **ADR-0031 の実装に決定↔前提の乖離（RV-9）**。単調増加の baseline をグローバル最大タグ（`git tag --list` の最大版・`check-release-invariants.mjs:32-40,96`）にしたため、git-flow の「release を main に直マージ→自動タグ→**手動** back-merge」（runbook §2・ADR-0030）の順で **タグ作成〜back-merge の窓で develop の version が最新タグを下回り**、その間 develop への通常 PR が `compareSemver(version, baseline) < 0`（`:84`）＝「版の逆行」で落ちる。ADR-0031 が前提にした「通常 PR では version==baseline ＝毎 PR 安全」が back-merge ラグを見落としている。さらに `ci.yml` は `fetch-tags: true`（shallow）なので「HEAD から到達可能な最大タグ」での判定（＝git-flow 的に正しい baseline）には履歴が足りず、グローバル最大に倒さざるを得ない構造。現状 develop=0.2.0＝最新タグ v0.2.0 で**未発火（latent）**だが、次回リリースの back-merge 窓で表面化する。推奨は (a) 到達可能最大タグ＋`fetch-depth:0`、(b) 検査を base=main の PR に限定、(c) ADR amend で back-merge 即時必須化 のいずれか（baseline 定義に関わるため ADR-0031 の amend を伴う）。
