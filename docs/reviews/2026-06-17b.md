### 1. API 忠実性 — 🟡 RV-5（RV-1 は fixed 検証）

**RV-1（fixed・検証）**: ADR-0020（案A+2a+3a+軸4）が実装に正しく落ちている。`src/resources/resource.ts:127-132` の `defaultFieldList` がカタログから既定 field を生成し、User 型は 4 サブ項目（`User.P_Id/P_Type/P_Name/P_Mail`）を入れ子展開、System[Reference] は `()` 省略で ID のみ＝`decode.ts` の `decodeUser`/`decodeReference` と 1:1 対応。`search` は `field: query.field ?? defaultFields`（`:229`）で、**省略=既定送信／`[]`=主キーのみ（`buildReadUrl:172` が空配列を送らない）／明示=verbatim** の 3 系統を実現。`get` も内部 `search` 経由で既定 field を得る。Attachment は bespoke で `DEFAULT_FIELDS`（メタデータのみ・`Content` 除外）。READ クエリは `URLSearchParams` で URL エンコード（reference 要件）。これで `one?.P_Name`（`README.md:63`）が実機でも成立する。

**Option 複数選択（再確認・良好）**: `decode.ts:82-91` の `decodeOption` は `Object.keys(root)` で**選択された全 alias を `string[]` で返す**（ADR-0017）。過去メモにあった「複数選択で先頭のみ＝データ損失」は解消済み。

**🟡 RV-5（新規）**: 送信前のリクエスト長ガードは `src/http/requester.ts:75-87` で **`req.body` の長さのみ**を見る。GET（Read）は body を持たず、`field`/`condition` は URL に載るため、Read 経路は長さガードの対象外。ADR-0020 で field 省略時に既定 field を全送信するようになり Read URL が伸びた（利用者が大きな明示 `field`/`condition` を渡せば一段伸長）。正典は「**リクエスト全体の長さ** ~15000 文字」（`docs/reference/resource-api/README.md:86`）＝GET では URL 全体が対象。超過すると、ライブラリが約束する明確な `PortersConfigError`（CLAUDE.md 固有注意点 5／README「送信前に弾く」）ではなく、不透明な server 400 に倒れる。加えて ADR-0020 の Consequences(Neutral)（`0020-…md:83`）は「リクエスト長超過時は既存の送信前ガードが働く」と記すが、その Read 経路は実際には未カバー＝**ADR の記述と実装が食い違う**。

なぜ今 latent か: 現行 MVP カタログは小さい（最大 Job=34 項目）。既定 Read URL は概算 ~1.2KB で 15000 に遠く、実害は出ない。だが「中核フェイルセーフに非対称な穴があり、ADR がそれを覆っていると誤記している」点は、カタログ拡張・count API 追加・巨大 field 指定で顕在化し得るため、退行追跡として起票する（→ findings RV-5）。

### 2. アーキテクチャ / ADR 遵守 — 問題なし

層分離は維持（transport → requester → resources → xml）。`resources/resource.ts` は decode/encode を `xml/` から import し、XML 構築を自前で持たない。ADR-0020 の既定 field 生成は `defaultFieldList` としてカタログ駆動（ADR-0019 の SoT を一段広げる設計どおり）。Attachment（`attachment.ts`）は ADR-0018 の bespoke 設計どおり：XML タグ組み立てを内包するが、値のエスケープは `xml/encode` の `encodeField` に委譲しており、注入面の責務は xml 層に残る（タグ名は定数のみ）＝許容範囲。

## 良かった点（strengths）

- **RV-1 修正の質**: カタログ → 既定 field（User 4 サブ展開・Reference ID-only）を decode 契約と 1:1 で生成し、`field: []` で API ネイティブ挙動も到達可能に残す＝薄いラッパーの fidelity とフェイルセーフを両立。透明化（README/JSDoc 明記）も実施。

- **不確実性の管理継続**: Attachment の既定項目・get 条件は LV-3/LV-4 として VERIFY(live) で追跡（契約待ち）。
