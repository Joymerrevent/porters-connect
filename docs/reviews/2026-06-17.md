## 観点別所見

### 🔴 1. API 忠実性 — RV-1（最重要）

正典 `tmp/porters-docs/txt/115010010367-…緩和措置あり.md` のとおり、**2015-07-01 以降に発行された App ID（＝現行すべて）** では Read で `field` 未指定だと `{Resource}.P_Id` **のみ**返る（緩和措置は旧 App 限定）。

しかし `src/resources/resource.ts:142` の `buildReadUrl` は `field` が空なら送らず、`:228` の `get(id)` は field を渡す手段が無い。結果 `porters.candidate.get(10001)` は `{ P_Id: 42 }` だけを返し、`one?.P_Name` は常に `undefined`。README クイックスタート（`README.md:62-63`）はこの実機では動かない挙動を提示している。`search({condition, count})`（field 省略, `README.md:55`）も同様。

**なぜ重要か**: 「基本的な read が空に見える」ため、このライブラリの第一価値（型付きレコードを返す）が既定経路で成立しない。実用ブロッカー級。

**推奨**: ADR-0019 でカタログが single source of truth になった今、`field` 省略時は**カタログ全 `P_` alias を既定送信**するのが最もフェイルセーフ。`get` も内部で全 field 要求に。仕様変更が重いので ADR 化を検討（既定 field の決め方／custom `U_`/`A_` の扱い）。

### 🟡 6. テスト厳密性 — RV-2

`src/resources/candidate.test.ts:65` は field 無し `search()` に対し全項目を返す `ALL` fixture（`:27-41`）で full decode を検証している。これは RV-1 の本番挙動（P_Id のみ）と食い違う「あり得ない応答」を前提にしており、罠を隠す。`get(7)` テスト（`:103-109`）は 0 件応答のみで field を検証していない。RV-1 を直す際、fixture を現実形に直し、既定 field 注入を pin するテストを追加する。

### 🟡 5. エラーモデル — RV-3

`src/errors/porters-error.ts:11` の `ErrorCategory` に `"rateLimit"` があるが、`src/errors/classify.ts` の `resourceCategory`/`authCategory` は一度も返さない。PORTERS はレート超過時に 429 ではなく強制切断するため、実際は `PortersNetworkError`（category `"network"`、`fetch-transport.ts:29`）に倒れる。`PortersNetworkError` の JSDoc も「forced rate-limit disconnect」と明言。よって `"rateLimit"` は到達不能で、`README.md:229` の `e.category` 例と不整合。予約注記／型削除／強制切断検知への配線のいずれかで整合を。

### 🟡 8. リリース準備 — RV-4

`@joymerrevent/...` はスコープ付き＝npm 既定 private。`publishConfig.access:"public"` が無いと `npm publish` が失敗する。`repository`/`bugs`/`homepage`/`keywords`/`author` も未設定（発見性・帰属のため公開前に推奨）。version `0.0.0` は未公開なので妥当。

### 2. アーキテクチャ / ADR 遵守 — 問題なし

層分離が明確（transport → requester → resources → xml）。XML は `resources/` の外（`xml/` に集約）に保たれ、第1層に業務ロジックの混入なし。最近 accept の ADR-0019（カタログ駆動の静的型）が実装に正しく反映。

### 3. コーディング規約（ADR-0013）— 問題なし

factory 関数で契約型を返す、class は Error 派生と `PortersClient` のみ、関数は全 arrow、型は全 `type`。generator 例外は `resource.ts:213` でコメント説明。`src/index.ts` は明示 export でキュレーション。lint 緑。

### 4. 型安全 / 公開面 — 問題なし

`ReadRecord` が全項目 optional+nullable で field 選択モデルを型で正確に表現（RV-1 の挙動とも論理整合＝だからこそ既定 field が要る）。`expectTypeOf` の型レベルテストで契約を pin。custom `U_`/`A_` 非型付けは ADR-0005 SD-2 として明示的に先送り済み。

### 7. ドキュメント / DX — 概ね良好

README に非公式明示・契約必須注記・Connect API Version 2 明記あり（CLAUDE.md「省かない」を満たす）。`live-verification.md` の VERIFY(live) ↔ 表の対応が保たれ、LV-5 は 2026-06-17 に確定済み。唯一 RV-1 関連の get/P_Name 例が実機と乖離。

## 良かった点（strengths）

- **フェイルセーフの具現化**: 送信前サイズガードで throttle/token を浪費しない（`requester.ts:75`）、非冪等 create のネットワーク不確実時 surface（`requester.ts:112`）、401/402 reactive refresh、未知コードは握り潰さず `"unknown"`。
- **カタログ駆動型（ADR-0019）**: 同一 `as const` カタログから静的型と実行時 lookup を両取り。
