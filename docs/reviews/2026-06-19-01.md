## 観点別所見

固定 8 観点を番号順に通す。指摘ゼロの観点も退行検知のため記録する。

### 1. API 忠実性 — 問題なし（観察 1 件）

R-16/R-17 に API 忠実性のブロッカーは無い。

- **R-16 値エンコード**: 宣言したカスタム項目は既存のカタログ駆動経路（`decodeField`/`encodeField`/`defaultFieldList`）にそのまま乗るため、Option=`string[]`、DateTime/Date=ISO⇄PORTERS、User=ID 書き戻し の round-trip は標準 `P_` と同一機構で担保される。`{...FIELDS, ...custom}` のマージは `U_`/`A_` が `P_` と衝突しないため `as typeof FIELDS & C` が健全。
- **R-17 認証自動応答**: モックの token XML（`AccessTokenExpiresIn=1800000` 等＝ms）は `token-provider` の `now()+expiresIn`（ms 前提）と整合（CLAUDE.md「単位は ms」・`docs/reference/authentication-api/token.md`）。`/v1/oauth`・`/v1/token` の substring は MVP の全リソースパスと衝突しない。
- **観察（非指摘）**: ビルダー `f.user()` は custom `U_`/`A_` の User 型を許すが、テナント固有の User 型カスタム項目の wire 形は live 未検証（標準 User 経路を再利用するため実害は低い）。Reference/Image は ADR-0023 D3 で意図的に対象外。custom User 型を本番採用する段階で LV 追加を検討。

### 2. アーキテクチャ / ADR 遵守 — 🟢 RV-8

層分離は保たれている（mock transport は `http/`＝トランスポート層、DSL は `fields/`、XML は `xml/` に閉じ `resources/` へ漏れない）。ADR-0023（D1 ジェネリック client / D4 検証境界 `defineFields` / D6 データ 5 種限定）と ADR-0024（D1 ハンドラ核 / D2 認証自動 / D4 メインエントリ / D6 MSW 非軸 / D7 フェイクサーバー follow-up）は実装に忠実。唯一、`EmptyCatalog` を `resources/*` が `../fields` から取るため依存が resources→fields と上向き（型のみ・循環無し）。緊急ではないが、`read-core` へ移すと方向が揃う（RV-8）。

### 3. コーディング規約（ADR-0013） — 問題なし

### 4. 型安全 / 公開サーフェス — 問題なし

## 良かった点（strengths）

- **seam を狙う設計が一貫**: `createMockTransport` を `fetch` でなく公開 `Transport` seam に接地（ADR-0024 D6）＝内部実装変更に強い。評価レバーの土台が薄く堅い。
- **既存機構の再利用**: R-16 はカタログ駆動の型/decode/encode をそのまま使い、コア追加が薄い（新並行経路を作らない）。
- **フェイルセーフの徹底**: `defineFields`/未モック route とも黙殺せず判別可能エラーで即時 surface。R-18 が「自動回復で直らなかったときだけ表に出る」前提を明文化。
- **ADR 規律**: R-16/R-17 とも proposed→議論→accepted→実装→PR を踏み、決定が実装に追従。R-18 は ADR 不要（ADR-0006 接地のドキュメント）と適切に判断。
