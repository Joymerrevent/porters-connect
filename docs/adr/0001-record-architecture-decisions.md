# 1. ADR で設計判断を記録する

- ステータス: Accepted
- 日付: 2026-06-12
- 関連: `SPEC_v1.md`, `CLAUDE.md`

## コンテキスト

`SPEC_v1.md` は調査ベースの**素案**で、主要な設計判断（HTTP・エラー型・OAuth ライフサイクル・XML→型・公開 API など）はまだ議論・確定していない。
本ライブラリの哲学は**フェイルセーフ**＝「壊れたときに安全側へ倒れるものを作る」。長く保守し、将来は外部貢献者も迎える前提なので、「なぜその設計にしたか」を残す仕組みが要る。

## 決定

設計判断を ADR（Architecture Decision Record）として `docs/adr/` に 1 判断 1 ファイルで記録する。
運用ルールは [`README.md`](README.md) に従う。

## 理由

- コードやコミットログには「選ばなかった案とその理由」が残らない。ADR はそこを補完する。
- 1 判断 1 ファイルなので、論点を分割して 1 つずつ合意形成できる。
- 軽量（Markdown のみ）で、既存の lint/format/CI パイプラインにそのまま乗る。

## 検討した他の案

- **SPEC を直接書き換え続ける**: 履歴に「なぜ変えたか」が残らず、議論の経緯が消える。
- **Issue/PR の議論だけで管理**: リポジトリ単体で完結せず、後から追いにくい。

## 結果（トレードオフ）

- 得るもの: 設計の透明性・再現性、議論の単位化。
- 払うコスト: ADR を書く一手間。ただし雛形で軽減する。
- 見直し条件: 運用が重いと感じたら README の運用ルールを ADR で更新する。
