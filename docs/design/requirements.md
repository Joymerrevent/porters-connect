# 要件定義（PRD）— @joymerrevent/porters-connect 第1層 v1

- ステータス: draft
- 日付: 2026-06-13
- オーナー: jun.shiromoto (Joymerrevent)
- 位置づけ: `SPEC_v1.md` は素案（superseded 予定）。設計の正は本 PRD・[docs/adr][adr]・[docs/reference][ref]。
- 対象: **第1層（TypeScript ラッパー）の v1**。第2層 MCP・第3層配布(Docker)・CLI は対象外（将来）。

## 1. 課題（Problem Statement）

PORTERS Connect API は**レスポンスが XML 専用・OAuth が独自仕様・UTC 前提・削除 API なし・レート/サイズ制限あり・項目がテナント毎に動的**と癖が強く、連携開発のたびに各社が `fetch` + XML パースを手書きしている（再利用可能な TS/JS ラッパーは公式・非公式とも存在しない）。
結果、**実装コスト・バグ・保守負債**が連携案件ごとに発生し、ポーターズ社にも「利用者コード起因の問い合わせ」が回る。

## 2. 北極星とゴール（Goals）

**北極星: デファクト（第一想起）** — 「PORTERS Cloud API で開発するなら porters-connect を使うのが常識」になること。利用者“数”ではなく“第一想起”を成功と定義する。

ゴール（成果ベース）:

1. **連携実装の立ち上げを劇的に短縮**: 認証〜最初の型付き取得までを「ドキュメントを見ながら数十分」で到達できる。
2. **公認に値する品質**: ポーターズ社が「紹介してよい」と思える＝良き API 市民・秘匿情報非漏洩・サポート負荷を増やさない。
3. **契約前でも評価できる**: 実 API 契約が無くてもモックで動作確認・学習・テストができ、採用検討の入口に立てる。
4. **壊れにくく、壊れても安全**: フェイルセーフ（想定外でも被害を広げない）を既定動作にする。
5. **受託・社内開発の現場で実際に採用される**: ブログ流入 → 試用 → 実案件採用、の導線が成立する。

## 3. 非ゴール（Non-Goals）

- **第2層 MCP サーバー**: 価値は大きいが第1層安定後（別フェーズ）。v1 では設計余地だけ残す。
- **第3層 配布（Docker / npx 専用体験）・CLI**: 後続。
- **実行時 i18n（多言語ロケール切替）**: バックエンドが日本縛りのため過剰。公開サーフェスの英語化のみ行う。
- **削除機能**: API に存在しない。`delete()` を型レベルでも生やさない。
- **MVP 外リソース**（Recruiter / Contact / Activity / Contract / Sales / Opportunity / Phase の Write 等）: v0.2 以降で段階追加。
- **GUI・ホスティング型サービス・スクレイピング**: スコープ外。
- **PORTERS 公式 SDK の置換主張**: あくまで**非公式**ラッパー。

## 4. 利用者像（Personas）

- **P1 受託開発者 / フリーランス / 制作会社（主）**: 人材会社から API 連携を受託。短納期・型安全・自己解決できるエラーを重視。**国内・日本語**。
- **P2 人材会社の社内エンジニア（副）**: 自社サイト⇔PORTERS 連携を内製。継続保守する立場。
- **（将来）AI エージェント**: 第2層 MCP 経由。v1 では視野に入れるが対象外。

## 5. ユーザーストーリー（User Stories）

受託開発者（P1）:

- 受託開発者として、**`PortersClient` を host/appId/appSecret/scope で初期化**したい。XML や OAuth の独自仕様を意識せず使い始めるため。
- 受託開発者として、**`porters.candidate.search(条件)` / `get(id)` が型付きオブジェクトで返って**ほしい。XML パースを書かずに済むため。
- 受託開発者として、**作成/更新を型付き入力で行い、XML を自分で組み立てたくない**。`P_Id=-1`/更新の作法やネスト XML を隠してほしいため。
- 受託開発者として、**200 件超の取得を自動ページングで**取りたい。1 リクエスト 200 件制限を意識せず全件処理するため。
- 受託開発者として、**失敗時に「何が・なぜ・どう直すか」が分かる型付きエラー**がほしい。ポーターズに問い合わせず自己解決するため。
- 受託開発者として、**契約環境が無くてもモックで動かして学習・テスト**したい。採用可否を判断するため。

社内エンジニア（P2）:

- 社内エンジニアとして、**レート制限・リトライ・サイズ制限をライブラリが面倒を見て**ほしい。深夜バッチが API を痛めず安全に回るため。
- 社内エンジニアとして、**日時が ISO 8601（タイムゾーン明示）で曖昧さなく**返ってほしい。自分のタイムゾーン表示は自前で行うため。
- 社内エンジニアとして、**テナント固有のカスタム項目（`U_`/`A_`）も扱える**ようにしたい。自社のカスタム項目を読み書きするため。
- 社内エンジニアとして、**対応 PORTERS / API バージョンが明記**されていてほしい。アップデート追従を判断するため。

## 6. 要件（Requirements）

### Must-Have（P0）— これが無いと v1 として成立しない

- **R-1 認証（OAuth）**: `code_direct`（サーバ間）でのトークン取得、Access Token のキャッシュと**自動リフレッシュ**、Refresh 失効時の明確な再認証エラー。初回の `code`（ブラウザ権限付与）は前提手順として**ドキュメント化**。トークンは差し替え可能なストア（既定インメモリ）。
  - 受入: 有効な資格情報で `search` が成功／Access Token 失効時に自動更新され処理継続／Refresh 失効時は判別可能な再認証エラーを返す。
- **R-2 型付きクライアント**: `new PortersClient({ host, appId, appSecret, scopes, ... })`。公開 API は `src/index.ts` から明示 export。
- **R-3 リソース操作（MVP）**: Candidate / Job / Client / Process / Resume / Attachment の **Read（search/get）と Write（create/update）**、および必要なマスタ Read（Partition / User / Field / Option）。**delete は提供しない**。
  - 受入: 各リソースで型付き search/get/create/update が動作（モック XML で検証）。削除メソッドは型に存在しない。
- **R-4 XML の隠蔽**: レスポンス XML → 型付きオブジェクトへ内部変換。Write は型付き入力 → XML を内部生成。**XML を外に漏らさない**。Option/参照/User/Link/Image のネストを正規化。
- **R-5 検索条件の型付き表現**: `field` / `condition`（型別 suffix）/ `order` / `keywords` / `itemstate` / ページング（`start`/`count`）を型で表現。
- **R-6 自動ページング**: `Total`/`Count`/`Start` を用い、200 件刻みで全件・逐次取得するヘルパー（暴走防止の上限/イテレータ）。
- **R-7 レート市民 & リトライ（フェイルセーフ）**: 1 分 Read 2000 / Write 500 を**自前スロットリング**で超えない（429 は返らない前提）。リトライ可能（Result Code `9`/`302`・ネットワーク）のみ指数バックオフで再試行、入力エラー（`100`系）は再試行しない。
  - 受入: 上限超で自動的に待機・分散する／一過性エラーは再試行し、確定的エラーは即座に判別可能エラーを返す。
- **R-8 リクエストサイズガード**: 全体長 ~15000 字を超える送信を**事前検知して警告/分割**（将来 16KB 化に追従可能に）。
- **R-9 構造化エラー（判別可能 union）**: 認証系とリソース系の**2 系統**を吸収。`category`（auth / permission / rateLimit / validation / network / transient / unknown ほか。**最終分類は ADR-0006 の 11 種**）・PORTERS の生コード・対処ヒントを持つ。**既定メッセージは英語**、ローカライズは利用側に委ねる。
  - 受入: 例外型/結果型から `category` と原因コードで分岐できる。
- **R-10 日時の表現（ISO 8601・タイムゾーン非変換）**: PORTERS の `yyyy/mm/dd HH:MM:SS`（UTC 扱い）/ `yyyy/mm/dd`（日付）を
  **ISO 8601 に正規化して入出力**する（DateTime は UTC 明示＝`...Z`、Date は日付のみで `Z` を付けない）。
  **JST 等の業務タイムゾーン変換は行わない（利用側の責務）**。Write でオフセット付き ISO を受けたら UTC に正規化して PORTERS 形式へ。`util/datetime` は「PORTERS 形式 ⇄ ISO の正規化」を担う。
  - 受入: 取得した DateTime が `...Z` の ISO で返る／オフセット付き ISO を渡しても UTC 化されて送信される／日付のみ項目は日付文字列で扱える。
- **R-11 秘匿情報の非漏洩**: App ID/Secret/トークンを**ログ・エラー・スナップショットに出さない**。設定は環境変数/引数から受け取り、リポジトリにコミットしない（`.env.example` のみ）。
- **R-12 差し替え可能なトランスポート（モック）**: HTTP 層を注入可能にし、**契約なしでもモック XML で全機能をテスト/評価**できる。ライブラリ自身のテストもこれで行う。
- **R-13 型安全の徹底**: 公開サーフェスに `any` を撒かない。リソース・スコープ・レスポンスを型で表現。
- **R-14 ランタイム/配布**: TypeScript(strict) / ESM / Node 18+ / MIT。`X-P-ConnectAPI-Version: 2` を既定送信し、**対応 PORTERS / API バージョンを README・コードに明記**。
- **R-15 言語方針**: 公開サーフェス（型名・メソッド名・public JSDoc・エラー既定文）は英語、内部コメントは日本語可。**README/docs は日本語ファースト**。

### Nice-to-Have（P1）— あると強いが核は無くても回る

- **R-16 動的カスタム項目**: 標準 `P_` は同梱の静的型、テナント固有の `U_`/`A_` は**利用者がフィールド定義を宣言 → 型導出＋実行時検証**で型安全に扱う（ハイブリッド）。`Field Read` からの宣言生成は将来の opt-in dev ツール。方式は ADR-0004 で決定。
- **R-17 評価用サンドボックス**: 契約前の評価を狙った、すぐ動く offline クイックスタート/サンプル（モック前提）。GTM の評価可能性レバー。
- **R-18 自己解決を促すドキュメント**: 日本語の「よくあるエラーと対処」（Result Code・認証エラーの 2 系統対応表つき）。サポート肩代わり。

### Future Considerations（P2）— v1 では作らないが設計で潰さない

- 第2層 MCP（公開ツール `search_candidates` 等）。第1層を内部呼び出しするだけにできる API 設計にしておく。
- MVP 外リソースの全 R/W、CJS 出力、CLI、Docker 配布、公開プレイグラウンド。

## 7. 成功指標（Success Metrics）※数値目標はベースライン取得後に設定

市場が小さい（実需は数十〜百数十社規模の推測）ため、**量より“デファクトの兆候”をマイルストーンで**測る。
ベースライン（実利用）が無い段階では数値目標を固定せず、達成シグナルで評価する。

先行指標（数日〜数週で動く）:

- **Time-to-first-typed-call**: `npm install` → 最初の型付き取得まで（クイックスタート通読で到達。実測後に数値化・暫定 30 分以内）。
- **評価可能性**: モック・クイックスタートが**契約なしで動く**（YES/NO）。
- **ドキュメント自己解決率**: 「よくあるエラー」で解決できる主要エラーの網羅（定性）。

遅行指標（数週〜数ヶ月）:

- **実案件採用数**: 受託/社内の実連携での採用。まず「**実案件 1 件目の採用**」をマイルストーンに（件数目標は実績後）。
- **流入→試用**: 「PORTERS API TypeScript」等での検索流入とリポジトリ到達（ブログ経由）。
- **ベンダー言及/紹介**: ポーターズ社からの言及・紹介ページ掲載・認定パートナー的位置づけ（0/1 で大きい）。
- **品質シグナル**: 実利用者からの Issue/PR の質（数より「本物の連携者か」）。

## 9. フェーズ計画（Timeline / Phasing）

実装は MVP リソース順に縦切りで進める（各段でモック XML テスト）。

1. **PoC**: OAuth（code_direct）＋ Candidate Read を型付きで実証（モック）。コア層（HTTP/XML/エラー/スロットリング/日時）の最小実装。
2. **MVP コア**: Candidate の Write、Job / Client / Process / Resume、Attachment（ファイル）。自動ページング・サイズガード・構造化エラーを通す。
3. **仕上げ**: 日本語 docs（よくあるエラー）・サンドボックス（R-17）・カスタム項目（R-16）・README（対応バージョン/非公式/契約必須）。

> 実 API 接続は契約環境が用意できてから PoC で検証。それまでは [docs/reference][ref] の XML 例を fixture にする。
